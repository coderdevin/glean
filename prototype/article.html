§ 1

Today, Durable Objects' replication topology — which we've tested internally for eighteen months — is leaving beta. Cross-region replicas are now available to every Workers paid customer; no allowlist, no waitlist. If you've ever asked "why does my chat app feel sluggish for Asia-Pacific users when the leader region is in Iowa?", this release is for you.

今天，我们内部测试了 18 个月的 Durable Objects 复制拓扑正式离开 beta——跨区域副本对所有 Workers 付费用户全量开放，没有 allowlist，也没有 waitlist。如果你曾经问过"为什么我的聊天应用对亚太用户感觉迟钝，而 leader 区域却在 Iowa？"，这次更新就是为你而做。

Under the hood, Durable Objects now run a modified Raft consensus where the leader region is configurable per-object — not per-namespace. This means a chat application can route Asia-Pacific traffic to a Singapore leader while US-East users hit a Virginia leader, with each object's writes serialized at its own home region. The change is the technical core of this release; region-level Raft as conventionally implemented cannot do this kind of per-tenant sharding.

底层上，Durable Objects 现在运行一份魔改版 Raft，leader 区域按对象可配——不是按 namespace 可配。这意味着一个聊天应用可以把亚太流量路由到新加坡 leader，而美东用户访问 Virginia leader，每个对象的写都在自己的 home region 序列化。这是本次发布的技术核心；按 region 级别实现的传统 Raft 做不到这种 per-tenant 分片。

Reads happen at the nearest replica that has caught up within a freshness window you specify (default: 50 ms). For chat or rate-limit workloads where 50 ms of staleness is invisible to users, this means cross-region reads serve from the closest edge. For billing or finance workloads where a strict read-after-write guarantee is non-negotiable, set the window to 0 and reads route to the leader — at the cost of round-trip latency. One configuration knob, dial your CAP tradeoff.

读发生在 最近的、已追上你指定的 freshness 窗口（默认 50ms）的副本上。对于聊天或限频这类 50ms 的过期对用户不可感知的场景，跨区域的读可以从最近的 edge 服务。对于账务、金融这类严格 read-after-write 一致性不可妥协的场景，把窗口设为 0，读会路由到 leader——代价是 round-trip 延迟。一个参数旋钮，调节你的 CAP 权衡。

Existing applications require no code changes — only a single line in wrangler.toml: replicas = "global". Billing is calculated by the bytes of additional replica synchronization, roughly the sync bandwidth multiplied by the number of replicas. There is no separate "replication SKU" to purchase, no new dashboard to learn, no agent to deploy. Monday decide, Tuesday deploy, Wednesday see data.

已有应用不需要改任何代码——只需在 wrangler.toml 加一行：replicas = "global"。计费按"额外副本同步"的字节数计算，约等于同步带宽 × 副本数。没有单独的"副本 SKU"要购买，没有新的 dashboard 要学，没有 agent 要部署。周一决定，周二上线，周三看数据。

§ 5

Three workloads earn the most from cross-region replication: real-time collaboration where multi-region users share state (Figma-like editors, chat); counters and rate limits where reads dominate writes and reads can be slightly stale; and AI inference routing where requests dispatch to the nearest model endpoint based on the user's geographic location. Single-region D1 users, static Pages workloads, and KV-only services see no benefit and should keep their current configuration.

三类工作负载从跨区域复制中获益最大：实时协作场景，多区域用户共享状态（Figma 类协作工具、聊天）；计数器与限频场景，读远多于写、且读可以略微过期；AI 推理路由，请求根据用户地理位置 dispatch 到最近的模型端点。单区域 D1 用户、纯静态 Pages 工作负载、KV-only 服务从这次更新中得不到收益，应该保持当前配置。

§ 6

Caveats: replicas are only available on Workers paid plans; the free tier remains leader-only. Per-byte pricing is currently dashboard-only — the public price list is still being iterated. Cold-start latency for newly-promoted replicas is unpublished; our early-tester measurements put Asia-Pacific to US-West sync at 80–120 ms. A deep-dive on replica selection and conflict resolution follows in the next post.

几点注意：副本仅在 Workers 付费域可用；自由层仍然只走 leader 强一致。每字节的详细计费目前只在 dashboard 可见——公开价格表我们还在迭代。新晋升副本的冷启动延迟尚未公布；内部 early-tester 测得亚太到美西的同步在 80–120ms 之间。副本选择和冲突解决的深入讲解在下一篇文章。

公告没强调的解锁：metadata 模型变了 The unstated unlock: the metadata model changed

这次发布在博客里被定位为"跨区域副本 GA"，但更深的解锁是 leader region 配置从 namespace 级降到了 object 级。两者听上去差不多——实际是两种不同的产品。Namespace 级意味着"你的整个应用部署在某个 region 家族"；object 级意味着"你的应用里每个 DurableObject 实例可以选择自己的 home region"。后者把 routing 从部署决策变成了运行时决策。 The post frames this as "cross-region replicas GA," but the deeper unlock is moving leader-region configuration from namespace-scope to object-scope. These sound similar; they are different products. Namespace-scope means "your whole app lives in a region family." Object-scope means "every DurableObject instance can choose its own home region." The latter promotes routing from a deployment decision to a runtime decision.

代价是 control-plane metadata 体量随 object 数线性增长。Cloudflare 博客里没解释他们怎么 shard 这个 metadata——这是真正值得追问的工程问题，可能在下一篇 deep-dive 里揭示。如果你在做技术选型的判断，比起"50ms freshness 默认值是不是聪明"这种 surface 问题，问"per-object metadata 的 control-plane 容量在哪里被设计"——这是会决定产品 5 年命运的问题。 The cost is control-plane metadata that grows linearly with object count. The post doesn't explain how Cloudflare shards this metadata — this is the technical question worth pressing, likely answered in the next deep-dive. If you're making a vendor decision, asking "where is per-object metadata's control-plane capacity engineered" matters far more than surface questions like "is the 50ms freshness default smart." The former determines the product's 5-year fate.

G
GPT-5 OpenAI

"零代码改动"是真的，但有三笔账没写在博客里 "No code changes" is true; three costs are not on the post

"replicas = "global" 一行就够" 在工程上是真的，但读者不应该把这等同于"零成本"。三个隐藏 trade-off： "One replicas = "global" line suffices" is engineering-true, but readers shouldn't equate this with "zero cost." Three hidden trade-offs:

  1. Read-after-write 假设需要重审。单 leader 拓扑下，"写完立即读到自己"为真；切到 replicas = "global" 后，50ms freshness 窗口内不再为真。你的代码字面上不用改，但代码里隐含的逻辑假设要重新审计——尤其是任何 "POST 之后 GET 应该看到 POST 内容" 的端到端测试。 Read-after-write assumptions need re-audit. Under single-leader, "write then immediately read your own write" was true; under replicas = "global", it's not true within 50ms. Code doesn't change; assumptions embedded in code must be re-validated — especially end-to-end tests that assert "GET after POST sees POST."
  2. 计费按"额外副本同步字节数"——viral object 是定价炸弹。一个被全球频繁更新的对象（聊天群组、热门 feature flag），同步带宽可能是单 region 写入流量的 5-10 倍。账单可能比预期高一个数量级。 Pricing scales on "extra replica sync bytes" — viral objects are a billing landmine. A globally-and-frequently-written object (a viral chat room, a popular feature flag) generates 5-10x its single-region write bandwidth in sync traffic. The bill can be an order of magnitude over expectations.
  3. Hot-leader 风险没有自动缓解。Per-object granularity 让单个 viral object 可能让该 region 的 leader 过载——而 Cloudflare 没在博客里讲自动负载分散机制。建议在 staging 跑一个 hot-object 压测。 No automatic hot-leader mitigation. Per-object granularity means a single viral object can saturate its region's leader — and the post doesn't describe automatic load-shedding. Run a hot-object stress test in staging.

放进 Cloudflare 18 个月的剧本里——下一步是 D1 跨区域 In Cloudflare's 18-month script — D1 cross-region is next

Cloudflare 在 2024-2025 的 18 个月内做了一条清晰的"消除分布式应用认知开销"的路线：Workers RPC（跨服务直接调用）→ Smart Placement（自动选 region）→ 现在 Durable Objects 跨区域副本。每一步都是把"做分布式应用需要的脑容量"往边缘网络里下沉。 Cloudflare's 18 months (2024-2025) trace a clear "reduce the cognitive overhead of distributed apps" arc: Workers RPC (direct cross-service calls) → Smart Placement (auto-select region) → now Durable Objects cross-region replicas. Each step pushes "the mental load of doing distributed apps" down into the edge.
